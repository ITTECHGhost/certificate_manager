# The Cerebrum

## Architectural Rules
- Always use the predefined UI themes and color palettes defined in `config.py`.
- Keep database interaction logic within `db.py` or use centralized queries from `data/queries.py` instead of scattering SQL throughout UI files.
- UI components should use `customtkinter` (CTk) classes where possible to maintain the modern appearance.
- Follow the established standard for creating screens (using the `SidePanel` and `RecordList` patterns as seen in other screens).
- Ensure new data models and tables are correctly versioned/migrated if the schema changes.

## Bug Ledger
- **ImportError: `backup_db`/`restore_db` missing from `db.py`** (2026-05-12): After the MySQL migration, `backup_db` and `restore_db` were removed from `db.py` but `settings_screen.py` still imported them, crashing on startup. Fixed by re-implementing both functions in `db.py` using `subprocess` to call the `mysqldump` and `mysql` CLI tools. Credentials are passed via the `MYSQL_PWD` environment variable (never on the command line). If the tools are not on PATH, a descriptive `RuntimeError` is raised at call-time — the app never crashes on import. Also fixed a duplicate import block in `settings_screen.py` (the first ~9 lines were duplicated) and updated file dialogs to use `.sql` instead of the legacy `.db` SQLite extension.
- **Student search/edit broken — 6 legacy bare-function calls never migrated** (2026-05-12): `EnrollmentPanel` and `StudentsScreen._do_delete_student` still called the old `data.queries` bare functions (`get_courses_for_dept_stage`, `get_enrollments_for_period`, `insert_enrollment`, `update_enrollment`, `delete_enrollment`, `delete_student`) that no longer exist after the Repository Pattern migration. This caused NameErrors at runtime whenever a user tried to: (a) open the grades panel for a period, (b) add/edit/delete an enrollment, or (c) delete a student. Fixed by replacing all 6 calls with their correct Repository equivalents: `CourseRepository().get_by_dept_stage_system()`, `EnrollmentRepository().get_by_period()`, `.insert()`, `.update()`, `.delete()`, and `StudentRepository().delete()`.
- **Search crashes on NULL values in Student and Course screens** (2026-05-12): The search logic in `StudentsScreen` and `CoursesScreen` was directly calling `.lower()` on database fields like `full_name_en` or `dept_name_ar` without checking for `NULL` (`None` in Python). If a record had a missing field, the search would crash with an `AttributeError: 'NoneType' object has no attribute 'lower'`. Fixed by adding `(value or "")` guards to all search-related fields. Also improved `CourseRepository.get_all` to correctly aggregate shared department names using `GROUP_CONCAT` so that shared courses are searchable by department.
- **Search crash: Unknown column 's.graduation_date'** (2026-05-12): `StudentRepository.search()` selected `s.graduation_date` in its SQL, but that column does not exist on the `students` table (graduation info is stored on `graduation_orders`). Every search keystroke crashed with `mysql.connector.errors.ProgrammingError: 1054`. Fixed by removing the non-existent column from the SELECT list.
- **Schema mismatch: `passed_round`, `semester_weight`, `display_type` etc. never existed in DB** (2026-05-12): Multiple repository methods and UI components referenced columns that were planned but never added to the MySQL schema. `academic_periods` has no `passed_round` column; `study_systems` has no `semester_weight`, `year_weight`, `prefix`, `display_type`, `weight_type` — the real columns are `period_display` and `calculation_weights`. Fixed `AcademicPeriodRepository` (get_by_student, insert, update), `StudySystemRepository` (get_all, get_active, insert, update), `CertificateRepository.get_full_certificate_data`, `settings_screen.py` (UI and save logic), and `students_screen.py` (period card display).
